## О проекте

`myMarket` — Маркетплейс цифровых товаров.

Система включает публичный каталог, сценарий оформления заказа, кабинет продавца для управления товарами и цифровыми позициями, а также административную часть.

Проект охватывает полный жизненный цикл заказа:

1. Клиент выбирает товар и оформляет заказ.
2. Если выбранный вариант товара предполагает автовыдачу, система проверяет доступность цифровых позиций и резервирует их.
3. Фоновая задача формирует ссылку на оплату.
4. Webhook подтверждает оплату и переводит заказ в статус `paid`.
5. После подтверждения оплаты зарезервированные цифровые позиции закрепляются за покупателем.
6. Неоплаченные заказы автоматически отменяются по истечении установленного времени.
7. При отмене заказа зарезервированные цифровые позиции снова становятся доступны для продажи.

Клиентская часть реализована на Nuxt. Основное приложение построено на FastAPI. Для хранения данных используется PostgreSQL, для инфраструктурных компонентов и фоновой обработки — Redis и TaskIQ. Обработка изображений вынесена в отдельный сервис. Проект запускается в контейнерном окружении через Docker Compose и Nginx.

> **Статус:** проект находится в активной разработке. Платёжный поток уже реализован на уровне приложения: фоновая задача формирует ссылку на оплату, а webhook переводит заказ в статус `paid`. Внешний платёжный провайдер пока заменён тестовой реализацией.

## Ключевые сценарии реализации

### Заказ и выдача цифрового товара

**Ручная выдача.** Покупатель оформляет заказ и продолжает обсуждение покупки с продавцом в связанном с заказом диалоге.

**Автовыдача.** Для вариантов с автовыдачей продавец загружает пул цифровых ключей. При создании заказа система выбирает и резервирует доступную цифровую позицию.

**Защита от повторной продажи.** Резервирование выполняется с блокировкой строк в PostgreSQL, поэтому одна и та же цифровая позиция не может попасть в несколько заказов при одновременных покупках.

**Оплата и отмена.** Формирование ссылки на оплату выполняется в фоновой задаче. Webhook подтверждает оплату и переводит заказ в статус `paid`. Если заказ не был оплачен вовремя, плановая задача отменяет его и возвращает зарезервированную позицию в продажу.

**Спор по заказу.** Покупатель и продавец могут привлечь поддержку к диалогу, если не могут самостоятельно решить вопрос по покупке.

### Каталог, фильтрация и поиск

**Категории и характеристики.** Каталог построен на иерархии категорий. Для каждой категории задаётся собственный набор характеристик, используемый при создании вариантов товаров и формировании фильтров.

**Фасетная фильтрация.** Пользователь может комбинировать параметры выбранной категории. Система отображает подходящие товары и актуальные значения фильтров с учётом уже применённых условий.

**Поиск.** Поиск по товарам и категориям дополняется быстрыми подсказками, чтобы пользователь мог перейти к подходящему разделу каталога или карточке товара.

### Доступ и роли

* **Покупатель:** просматривает каталог, оформляет заказы и ведёт диалоги по покупкам.
* **Продавец:** управляет товарами, вариантами и цифровыми позициями.
* **Администратор:** работает с административными и системными сценариями.

## Устройство приложения

```text
Nuxt frontend
      │
      ▼
    Nginx
      │
      ▼
FastAPI application
  ├── PostgreSQL
  ├── Redis + TaskIQ
  │     ├── worker
  │     └── scheduler
  ├── Image service
  └── Payment adapter
          │
          └── webhook ──► FastAPI application
```

**Основное приложение.** FastAPI предоставляет API и реализует прикладные сценарии: каталог, заказы, авторизацию и работу с пользователями.

**Хранение и фоновые процессы.** PostgreSQL хранит данные каталога, пользователей, заказов и цифровых позиций. Redis используется брокером TaskIQ для фоновых и запланированных задач.

**Внешние сервисы.** Image service создаёт миниатюры загруженных изображений вне основного API. Payment adapter формирует ссылку на оплату, а подтверждение платежа поступает в приложение через webhook.

### Организация backend-кода

* `api/` - HTTP- и WebSocket-endpoints, Pydantic-схемы, обработка ошибок и проверка доступа.
* `services/` - прикладные сценарии: каталог, заказы, пользователи и авторизация.
* `domain/` - сущности, правила предметной области, события и исключения.
* `adapters/` - репозитории, файловое хранилище, Redis, HTTP-клиенты, платежи и WebSocket.
* `db/` - SQLAlchemy-модели, мапперы и работа с PostgreSQL.
* `tasks/` - фоновые и запланированные задачи.

Прикладные сервисы работают с доменными объектами, а SQLAlchemy-модели и доменные сущности связываются отдельным слоем маппинга.

## Запуск

Проект запускается в Docker Compose-окружении.

### Требования

* Docker Engine;
* Docker Compose;
* файл `.env` с параметрами окружения.

```bash
git clone <repository-url>
cd myMarket
docker compose up --build
```
